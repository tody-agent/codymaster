# Proposal: CodyMaster Context Backbone v5 — "Smart Spine"
Generated by: cm-brainstorm-idea
Date: 2026-03-30

## Why we're doing this

### Qualified Problem

**For:** AI agent operators using CodyMaster as their skill orchestration platform
**Who:** Need reliable, efficient context management across 65+ skills and long-running sessions
**The:** CodyMaster context infrastructure is a fragmented collection of flat files (.cm/CONTINUITY.md, learnings.json, decisions.json, skeleton.md)
**That:** Needs a unified "nervous system" connecting memory, skills, and resources with enforced token budgets
**Unlike:** Current state where each skill independently reads/writes flat files and agents must "know" which file contains what
**Our approach:** Build an MCP-native context layer with SQLite storage, FTS5 search, L0 indexes, context bus, and token budget enforcement

### Root Causes
1. **Technical:** Organic skill growth without shared infrastructure. Memory tiers 4-5 (qmd, CodeGraph) documented but never activated despite 1,146 source files.
2. **Architectural:** CONTINUITY.md serves as both session state AND inter-skill communication (overloaded). No central bus or query engine.
3. **Operational:** Agent-dependent protocol enforcement — if LLM skips memory update or re-indexing, system degrades silently.

### 9 Windows Analysis Summary

| | Past | Present | Future |
|---|---|---|---|
| **Super-system** | AI agents = single-shot Q&A | Multi-turn + MCP emerging. Context = bottleneck | Autonomous agents with persistent memory. Context = managed asset |
| **System** | Skills collection, grew organically | 65 skills, 3/5 memory tiers active, no bus, no semantic search | Unified context engine. Skills self-coordinate. Memory queryable |
| **Sub-system** | Each skill independent, no shared state | 21 skills share Load/Update protocol. Flat JSON/MD files | Context Bus + L0 indexes + Auto-indexing daemon + URI scheme |

### Impact if NOT addressed
- **Scalability:** 100+ skills with no bus = chaos
- **Token waste:** Linear growth with project size (no progressive loading for learnings/skeleton)
- **Reliability:** Silent data loss when LLM skips protocol steps
- **Competitive:** Engine-based systems (OpenViking) will outclass skill-only approaches

Build the "Smart Spine" MCP context server, starting with Option A's quick wins.

**Why C wins:**
1. Agent-universal — MCP is the one protocol ALL supported agents understand
2. SQLite + FTS5 = dramatically better than flat files, lighter than Viking
3. Token budget enforced at API layer — not dependent on LLM behavior
4. Consolidates existing mcp-bridge.js + dashboard into one unified server

## Implementation Roadmap

### Phase 1: Quick Wins (Week 1-2) — Option A foundations

**1.1 L0 Index Generation**
- Create `src/l0-indexer.ts` — generates L0 summaries for:
  - `.cm/learnings-index.md` (~100 tokens: ID + 1-line summary per learning)
  - `.cm/skeleton-index.md` (~500 tokens: module list + entry points only)
  - CONTINUITY.md "Abstract" header (2-3 lines at top)
- Auto-trigger in `cm-start` and after skill chain completion
- Token savings: ~4,000 tokens/session → ~700 tokens/session for memory loading

**1.2 Context Bus (File-based)**
- Create `.cm/context-bus.json` with schema:
  ```json
  {
    "session_id": "string",
    "pipeline": "string",
    "current_step": "string",
    "shared_context": {},
    "resource_state": {}
  }
  ```
- Update skill chain to write output locations to bus after each step
- Update skill chain to read previous step output from bus at start
- Affected files: `src/skill-chain.ts`, `src/continuity.ts`

**1.3 Token Budget Config**
- Create `.cm/token-budget.json` with allocation per category
- `src/continuity.ts` checks budget before loading — switches to L0 when over budget
- Soft enforcement (warn) in Phase 1, hard enforcement (truncate) in Phase 2

### Phase 2: MCP Context Server (Week 3-5)

**2.1 SQLite Storage Layer**
- Add `better-sqlite3` dependency
- Create `src/context-db.ts` — manages:
  - `learnings` table (replaces learnings.json) with FTS5 index
  - `decisions` table (replaces decisions.json) with FTS5 index
  - `skill_outputs` table (context bus state)
  - `indexes` table (L0/L1 cached summaries)
  - `token_budget` table (per-session consumption tracking)
- Migration script: JSON → SQLite (one-time, backward-compatible)

**2.2 MCP Context Server**
- Expand `scripts/mcp-bridge.js` → `src/mcp-context-server.ts`
- MCP Tools exposed:
  - `cm_query(scope, query)` — FTS5 search across learnings + decisions + skill outputs
  - `cm_resolve(uri, depth)` — resolve `cm://` URI at L0/L1/L2 depth
  - `cm_bus_read()` — current pipeline state
  - `cm_bus_write(event, data)` — publish skill completion
  - `cm_budget_check(category)` — remaining token budget
  - `cm_memory_decay()` — trigger TTL cleanup
  - `cm_index_refresh(target)` — regenerate L0 for specific resource
- Merge with existing dashboard server (single process on port 6969)

**2.3 cm:// URI Scheme**
- Implement URI resolver in `src/uri-resolver.ts`:
  ```
  cm://memory/working       → CONTINUITY.md (or SQLite session)
  cm://memory/learnings     → FTS5 query on learnings table
  cm://memory/learnings/L005 → Specific learning by ID
  cm://memory/decisions     → FTS5 query on decisions table
  cm://skills/cm-tdd        → SKILL.md content at requested depth
  cm://skills/cm-tdd/L0     → Index entry only (~50 tokens)
  cm://resources/skeleton   → skeleton-index.md (L0) or full skeleton (L2)
  cm://pipeline/current     → context bus state
  ```

**2.4 File Watcher Daemon**
- Integrate chokidar into MCP server process
- Watch `src/**/*.{ts,js,py}` → auto-regenerate skeleton L0
- Watch `.cm/memory/*.json` → auto-regenerate learnings L0
- Debounced (5s) to avoid thrashing

### Phase 3: Advanced Features (Week 6+, optional)

**3.1 Session Compression**
- `cm commit` CLI command → archive old session, extract new learnings/decisions via LLM
- Automatic after session idle > 30 min

**3.2 OpenViking Backend (Future)**
- If/when OpenViking embedded mode stabilizes, swap SQLite backend
- `src/context-db.ts` interface stays the same — only storage engine changes
- Get true vector semantic search, L0/L1/L2 auto-generation

**3.3 Skill Auto-Coordination**
- MCP event streaming for skill-to-skill triggers
- `cm_bus_subscribe(event)` → agent gets notified when dependent skill completes
