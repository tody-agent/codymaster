# 🧠 Brain Upgrade: SQLite vs OpenViking (CodyMaster v4.6)

Starting from version 4.6, CodyMaster introduces **OpenViking** as an advanced Storage Backend, replacing the default SQLite engine.

This document will explain **through vivid examples** why OpenViking represents a massive leap forward in the AI Agent's ability to "understand" and "remember", and how it solves the inherent limitations of the legacy system.

---

## 🛑 The SQLite Problem (Legacy System - v4.5 and below)

CodyMaster v4.5 utilized SQLite combined with FTS5 (Full-Text Search). While incredibly fast, it had one fatal flaw: **It only performed Keyword Matching**.

### Scenario 1: The Silent Async Bug

You (or the Agent) once spent 3 hours debugging an issue and recorded the learning into memory:
> **Saved Memory:** `"Forgot to add await to the processQueue() function causing the data to hit a race condition and drop sporadically."`

A month later, the Agent encounters a similar bug in a different component and begins searching:
**Agent Query (SQLite):** `cm_query "data missing sporadically no error"`

❌ **SQLite Result:** `0 results` (Not found).
**The Cause:** The mechanical words in the search query do not perfectly match the keywords stored in memory. The Agent suffers from "amnesia" and might spend another 3 hours repeating the exact same mistake.

---

## 🚀 The Power of OpenViking (v4.6+)

OpenViking solves this problem using **Semantic Coordinates (Vector Embeddings)**. Instead of matching letters, it translates both the old experience and the new query into points within a mathematical space (e.g., OpenAI's 1536-dimensional space). The more similar two concepts are, the closer their points are to each other.

### 🌟 Use Case 1: Self-Healing via "Fuzzy Memory" (Semantic Recall)

**Same scenario:** The Agent is dealing with inexplicable data loss.
**Agent Query (OpenViking):** `searchAll("data missing sporadically no error")`

✅ **OpenViking Result:**
> 1. *"Forgot to add await to the processQueue() function causing the data to hit a race condition and drop sporadically."* (**89% Similarity**)

**Why did it work?**
The Vector space understands that `"race condition"` and `"sporadically"` are conceptually very close within a debugging context. From there, the Agent immediately recalls the fix without needing to guess the exact keywords. CodyMaster naturally becomes surprisingly "smart".

### 🌟 Use Case 2: Broad Architectural Analysis (Architectural Context)

The biggest challenge for an AI Agent is working on massive codebases where the Context Window (token limit) is easily overwhelmed.

**Scenario:** The user requests: *"Redesign the session management system. Don't forget the security constraints we agreed upon last month."*

#### How SQLite v4.5 worked:
*   The Agent queries: `cm_query "session security decisions"`
*   The result returns 1 or 2 files containing exactly those words. The Agent might miss the decision `"auth tokens must have strict TTL and rotate on IP change"` because the exact keyword `"session"` doesn't appear.
*   The Fallout: Proposing a design that violates internal standards, or blowing out the Token Budget by manually loading dozens of documents to blindly search.

#### How OpenViking v4.6 works (L0/L1 Auto-generation):
OpenViking doesn't just do Vector Search; it features **Dynamic Summarization**.

1.  The Agent calls `VikingBackend.searchAll("security requirements for authentication flows")`
2.  Instead of returning all 50 architectural documents, Viking clusters the related decisions.
3.  The Agent then calls `VikingBackend.getL1Overview('decisions')`.
4.  OpenViking ***automatically*** returns an incredibly concise Overview (L1) that has already distilled the key points related to Authentication, silently generated by the Viking engine itself (Auto-generate L1).

✅ **Result:** The Agent grasps a sharp, big-picture view without noise, consuming only a few hundred tokens instead of 10,000.

### 🌟 Use Case 3: "Intelligence Intersection" Across Languages (Cross-Lingual Search)

CodyMaster targets international environments. Multi-lingual search is a nightmare with FTS5.

1.  The Thailand team logs a Decision: `"ไม่ควรใช้ caching กับข้อมูล realtime"` *(Do not use caching for realtime data)*.
2.  A month later, you (in English) ask the Agent: `"What is the policy on using cache for realtime data?"`

❌ **SQLite FTS5:** Totally blank.
✅ **OpenViking:** Because it uses multi-lingual Embedding models, it understands that the Thai phrase and the English query share the same coordinates in the semantic space. It returns exactly that Decision!

## How to activate the "Viking Brain"?

OpenViking is a standalone engine ([OpenViking GitHub](https://github.com/volcengine/OpenViking)). After upgrading CodyMaster to `v4.6`, it takes less than 2 minutes to "plug in the brain":

1. Install and run the Server:
```bash
pip install openviking --upgrade
openviking start  # Defaults to port 1933
```

2. Configure CodyMaster (`.cm/config.yaml`):
```yaml
storage:
  backend: viking
  viking:
    host: localhost
    port: 1933
    workspace: codymaster
```

That's it! CodyMaster now possesses the most advanced semantic memory system available.
