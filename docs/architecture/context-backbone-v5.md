# 🦴 Context Backbone v5 — Smart Spine

> **"Give every skill a shared nervous system."**

Context Backbone v5 replaces the fragmented JSON memory layer with a unified SQLite+FTS5 storage engine, a cm:// URI scheme for progressive content loading, an MCP context server exposing 7 tools, and a token budget that hard-caps context before it overflows.

---

## Why It Was Built

CodyMaster's original 5-tier memory had a coordination problem: skills couldn't see what other skills had produced. Each session started cold, learnings were linear JSON arrays with no semantic search, and context load decisions were left entirely to the agent.

The result: repeated grep/glob chains, token waste, and no way to query *"did the previous step find what we needed?"*

Smart Spine solves this with four interlocking systems.

---

## System 1 — SQLite + FTS5 Storage

**File:** `src/context-db.ts` | **DB path:** `.cm/context.db`

All persistent memory now lives in a single WAL-mode SQLite database:

| Table | Purpose |
|---|---|
| `learnings` + `learnings_fts` | Error patterns with BM25-ranked keyword search |
| `decisions` + `decisions_fts` | Architecture decisions with keyword search |
| `skill_outputs` | What each skill produced (per session/chain) |
| `indexes` | Cached L0/L1 content with staleness hash |
| `token_usage` | Per-session token cost accounting |

FTS5 virtual tables are kept in sync via `AFTER INSERT` / `AFTER DELETE` triggers — no manual sync required.

**Key APIs:**
```typescript
queryLearnings(dbPath, 'i18n locale', 'module', 10)   // FTS5 + scope filter
queryDecisions(dbPath, 'namespace split')              // BM25 ranked
insertLearning(dbPath, learning)                       // trigger auto-syncs FTS5
```

## System 2 — Progressive Loading (L0 / L1 / L2)

Memory is always loaded at the cheapest sufficient depth.

| Depth | Tokens | Content |
|---|---|---|
| **L0** | ~100–500 | Index only — IDs + 1-line summaries |
| **L1** | ~600–2000 | Overview — first 40 lines or structured summary |
| **L2** | Full | Complete content |

L0 indexes are pre-generated by `src/l0-indexer.ts` and written to `.cm/`:

- `.cm/learnings-index.md` — compact list of all active learnings
- `.cm/skeleton-index.md` — entry points, modules, config files only (from skeleton.md)

Indexes are auto-refreshed when `addLearning()` writes new entries and on-demand via `cm continuity index` or the `cm_index_refresh` MCP tool.

A file watcher (`src/file-watcher.ts`) uses chokidar to auto-refresh the learnings L0 index whenever `.cm/memory/learnings.json` changes on disk.

## System 3 — cm:// URI Scheme

**File:** `src/uri-resolver.ts`

Any skill can request memory content by URI instead of reading files directly. The resolver handles caching, depth selection, and fallbacks transparently.

```
cm://memory/working              → CONTINUITY.md (L0 = abstract, L2 = full)
cm://memory/learnings            → learnings-index.md (L0) or SQLite (L1/L2)
cm://memory/learnings/{id}       → specific learning by ID
cm://memory/decisions            → decisions index or full list
cm://skills/{name}               → SKILL.md at requested depth
cm://skills/{name}/L0            → front matter + description only (~50 tokens)
cm://resources/skeleton          → skeleton-index.md (L0/L1) or full skeleton.md
cm://resources/architecture      → architecture.mmd
cm://pipeline/current            → live context bus state
```

**Usage in skill prompts:**
```
Load context: cm://memory/learnings at L0 (budget: 650 tokens)
Load skill:   cm://skills/cm-tdd at L1
```

## System 4 — MCP Context Server

**File:** `src/mcp-context-server.ts`

A standalone stdio MCP server (JSON-RPC 2.0 with Content-Length framing) that exposes the full Smart Spine layer to Claude Desktop and any MCP-compatible client.

### Tools

| Tool | Purpose |
|---|---|
| `cm_query` | FTS5 search across learnings + decisions. Supports scope filter and result limit. |
| `cm_resolve` | Resolve any `cm://` URI at L0/L1/L2 depth. |
| `cm_bus_read` | Read the live context bus — current pipeline, step, and all skill outputs. |
| `cm_bus_write` | Publish a skill completion event to the bus for downstream skills to consume. |
| `cm_budget_check` | Check if loading `N` tokens for a category is within budget. |
| `cm_memory_decay` | Archive learnings whose TTL has expired. Supports `dry_run`. |
| `cm_index_refresh` | Regenerate L0 indexes (learnings / skeleton / all). |

### Setup

```bash
cm continuity mcp   # prints config snippet + binary path
```

Add the printed snippet to `~/Library/Application Support/Claude/claude_desktop_config.json`.

```json
{
  "mcpServers": {
    "cm-context": {
      "command": "node",
      "args": ["/path/to/dist/mcp-context-server.js", "--project", "/path/to/project"]
    }
  }
}
```

## System 5 — Token Budget

**File:** `src/token-budget.ts` | **Config:** `.cm/token-budget.json`

A 200k-token window is pre-allocated by category:

| Category | Soft Limit |
|---|---|
| `system_prompt` | 5,000 |
| `skill_index_L0` | 2,500 |
| `skill_active_full` | 5,000 |
| `memory_working` | 500 |
| `memory_learnings` | 650 |
| `codebase_skeleton` | 1,500 |
| `context_retrieval` | 10,000 |
| `conversation_history` | 30,000 |
| `generation_budget` | 144,850 |

The budget is enforced at load time — `checkBudget()` returns `allowed: false` with a remediation suggestion when soft limits are exceeded. Override limits by editing `.cm/token-budget.json`.

```bash
cm continuity budget   # show allocation table with current usage
```

## System 6 — Context Bus

**File:** `src/context-bus.ts` | **State file:** `.cm/context-bus.json`

When a skill chain runs, the bus tracks every step's output so later skills can build on earlier work without reading the filesystem.

```typescript
initBus(projectPath, 'feature-development', sessionId)
// → creates .cm/context-bus.json

updateBusStep(projectPath, 'cm-brainstorm-idea', {
  summary: 'Recommended Smart Spine',
  output_path: 'openspec/changes/context-backbone-v5/proposal.md',
  affected_files: ['openspec/...'],
})
// → downstream skill reads cm://pipeline/current to see this
```

```bash
cm continuity bus   # pretty-print current bus state
```

## Migration from JSON

Existing `learnings.json` and `decisions.json` can be migrated to SQLite with the one-time migration utility:

```typescript
import { migrateJsonToSqlite } from './src/migrate-json-to-sqlite';
const result = migrateJsonToSqlite('/path/to/project');
// → { learnings: { migrated: 12, skipped: 0 }, decisions: { migrated: 6, skipped: 0 } }
```

Both files are backed up as `.backup` before migration. JSON files are kept for backward compatibility.
